# 悬浮球窗口抖动优化方案

## 问题根源

三种状态（静止 → 对话框 → 卫星菜单）频繁 `setBounds`，窗口 resize、重定位、内容渲染三件事并行，就抖。

---

## 方案一：大透明窗口，内容裁切（推荐）

**思路：** 窗口始终保持最大可能尺寸，永远不 resize。三种状态的元素全部放在同一个大窗口里，用 CSS `transform` + `opacity` 控制显隐和位置。

- 优点：彻底消灭窗口级抖动，零 resize 调用
- 缺点：窗口覆盖面积偏大，但因为是透明的不影响点击穿透（可配合 `setIgnoreMouseEvents` 处理）
- 适合：对话框和卫星菜单不会超出屏幕太多

---

## 方案二：不透明窗口只改位置，内容自适应

**思路：** 窗口大小在三种状态中取一个固定值（比如都按最大的卫星菜单来），状态切换只改内容布局，不改窗口尺寸。

- 优点：实现简单
- 缺点：浪费区域，半透明背景时不够美观

---

## 方案三：分层窗口

**思路：** 悬浮球一个窗口（始终不变），对话框和卫星菜单各自独立窗口。状态切换时销毁/隐藏对应窗口，而非 resize 当前窗口。

- 优点：每个窗口生命周期简单，不互相干扰
- 缺点：多窗口之间的坐标对齐要算准，跨窗口通信多一层

---

## 方案四：动画遮丑

**思路：** 如果必须 resize，那就在 resize 的瞬间做遮挡/过渡：

- resize 前先给内容加一层毛玻璃遮罩或缩小到球的状态
- `setBounds` 一步到位（不要分步 `setPosition` + `setSize`，必须一次性 `setBounds`）
- resize 完成后再弹出新内容

本质是**把抖动藏在视觉过渡里**，用户看到的是「收起 → 移动 → 展开」而非裸的 resize。

---

## 方案五：`setShape` 配合大窗口（Windows）

**思路：** 窗口尺寸保持最大，但用 `win.setShape()` 限制实际可点击区域为当前状态需要的形状。相当于方案一的变体，但物理可交互区域也是精确的。

- 优点：鼠标事件穿透的区域完全精确
- 缺点：`setShape` 在 macOS 上行为不一致，主要适用于 Windows

---

## 实际建议

| 场景 | 推荐方案 |
|------|---------|
| 追求丝滑、不抖 | 方案一（大透明窗口） |
| 不想改架构、快速修 | 方案四（动画遮丑 + `setBounds` 一次性到位） |
| 跨平台、复杂布局 | 方案三（分层窗口） |

**方案一是根本解法，方案四是最低成本的补救。** 大多数悬浮球产品（macOS 的 Spotlight、Windows 的 PowerToys Run）走的都是方案一的思路。

---

## 当前落地（desktop-agent）

- 已采用**方案一**：展开态窗宽固定为「双面板槽 + 双 `panelGap` + 球簇」，工具行内**绝对定位**，球簇 `left` 不随收起归零。
- 左/右展开只改面板 `left`，**不因切换方向 resize 窗口**。
- 球与输入条间距：`OVERLAY_LAYOUT.panelGap`。
- 窗高：平常 = 菜单展开高度；对话随内容增高，上限 50vh（见 README）。
- 更完整的踩坑与对照表见项目根目录 `README.md` →「悬浮窗布局：难点与方案」。
